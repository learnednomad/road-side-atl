# Security & Reliability Audit ‚Äî RoadSide GA

## 1. Executive Summary

RoadSide GA is functional but carries one **critical** and several **high** severity issues that put real money and customer PII at risk. The most urgent problem is a hardcoded admin password (`admin123`) baked into the seed code combined with an admin email-verification bypass ‚Äî anyone who knows the convention can log in as a platform admin. Compounding this, **production is running on placeholder/test secrets** (Stripe, Resend, Google OAuth all unset), which simultaneously breaks payments and email *and* makes the Stripe webhook forgeable, while two background-job/migration defects mean providers may go unpaid and schema drift fails silently on every boot. The good news: most access-control mistakes are localized inconsistencies (a few missing ownership checks) and the database is not currently exposed to the internet. The dominant theme is **deployment/configuration drift and missing guardrails**, not deep architectural rot ‚Äî fixable in days, but several items need attention today.

## 2. Findings by Severity

### CRITICAL

#### C1. Hardcoded default admin password `admin123` seeded into production
- **Location:** `db/seed-base.ts:185-204`; `db/seed.ts:46-49` (always runs base seed); creds printed at `db/seed-demo.ts:204`; verification bypass at `lib/auth.config.ts:33`
- **Plain English:** The seed script creates `admin@roadsidega.com` and `ops@roadsidega.com` with the password `admin123` hardcoded in source. The base seed runs even in production (`SEED_DB=true`), and admins are explicitly exempted from email verification, so a guessed admin password grants immediate full access with no second gate.
- **Impact:** Full platform takeover ‚Äî all customer PII, bookings, payments, and admin-only routes. The credentials are literally printed in the repo.
- **Fix:** Remove the constant. Require an `ADMIN_PASSWORD` env var with no default and fail boot in production if unset, or generate a random password and print it once. **Immediately rotate** the passwords of both admin accounts in the live DB and delete the printed creds from `seed-demo.ts`.

### HIGH

#### H1. Primary web login has no rate limiting / brute-force protection
- **Location:** `lib/auth.config.ts:14-44` via `app/api/auth/[...nextauth]/route.ts`
- **Plain English:** The NextAuth credentials login is a plain `bcrypt.compare` with no attempt counter or lockout. It lives outside the Hono app, so the existing rate-limit middleware (which only protects the *mobile* login) never touches it. There is no Next.js edge middleware fronting it either.
- **Impact:** Unlimited online password guessing / credential stuffing against any account, including admins (who also skip email verification ‚Äî see C1).
- **Fix:** Add per-email + per-IP rate limiting and progressive lockout to the credentials path (DB/Redis-backed counter), and apply the same to forgot-password/reset to stop token grinding. Note: the existing in-memory limiter also collapses clients (see H7) ‚Äî fix that too or this guard will be global.

#### H2. Production running on placeholder Stripe / Resend / Google OAuth credentials
- **Location:** Live container env (mirrors `.env.example:5-13,30`); detector at `lib/env.ts:64-86` only warns
- **Plain English:** The live container holds literal placeholders: `STRIPE_SECRET_KEY=sk_test_xxx`, `pk_test_xxx`, `RESEND_API_KEY=re_your-resend-api-key`, `AUTH_GOOGLE_ID/SECRET=your-google-...`. The app self-detects this at boot and logs a WARNING ‚Äî but boots anyway.
- **Impact:** Card payments cannot complete (invalid key). Transactional + verification/magic-link emails fail, which means non-admin signup verification cannot complete (login throws `EmailNotVerified`). Google OAuth is broken. Live availability and trust failure.
- **Fix:** Populate real production secrets in Coolify and redeploy. Change `lib/env.ts` to `process.exit(1)` (not warn) when known placeholders are present and `NODE_ENV=production`.

#### H3. Stripe webhook signing secret is the public placeholder `whsec_xxx` ‚Äî webhooks are forgeable
- **Location:** Live env `STRIPE_WEBHOOK_SECRET=whsec_xxx`; consumed at `server/api/routes/webhooks.ts:95-103,150-166`; route mounted `server/api/index.ts:120`
- **Plain English:** The only authentication on the Stripe webhook endpoint is an HMAC signature check keyed on a secret that is publicly published in `.env.example`. Anyone can compute a valid `Stripe-Signature` and POST forged events.
- **Impact:** A forged `checkout.session.completed` sets `payments.status='confirmed'`, overwrites `bookings.finalPrice` from attacker-controlled `amount_total`, and triggers `createPayoutIfEligible` ‚Äî i.e. payment fraud / unintended provider payouts. *Currently* partially contained because Stripe is otherwise disabled (H2) so no `stripeSessionId` rows exist ‚Äî but the `finalPrice` overwrite + payout trigger on a known `bookingId` are not gated on a matching payment row and are exploitable now; full fraud opens the instant a real Stripe key is set.
- **Fix:** Set the real webhook secret from the Stripe dashboard. Add a startup guard that hard-fails in production if the secret equals the placeholder.

#### H4. Cron jobs and WebSocket server never start in production
- **Location:** `docker-entrypoint.sh:64` (`exec node server.js`); `server-custom.ts:16-18`; `server/cron.ts`; `Dockerfile:82`
- **Plain English:** The entrypoint runs the Next.js standalone server (`server.js`), but cron scheduling and the WebSocket server are only started in `server-custom.ts`, which is compiled but **never executed**. There is no `instrumentation.ts` hook and no separate worker container.
- **Impact:** All 13+ background jobs are dead in production: `process-pending-payouts` (providers never get paid via manual batch), payment‚Üîpayout and Checkr/Stripe-Connect reconciliation, `offer-expiry-check` (bookings can stick in `dispatched`), onboarding deadline enforcement. All real-time WS updates silently no-op.
- **Fix:** Either `exec node dist/server-custom.js` in the entrypoint (and ensure `dist` is copied into the runner image), or move cron/WS startup into a Next `instrumentation.ts register()` hook with a singleton guard. Add a healthcheck that confirms the cron heartbeat.

#### H5. Database migrations fail silently on every boot
- **Location:** `docker-entrypoint.sh:53-58`
- **Plain English:** The entrypoint wraps `drizzle-kit migrate` in an `if/else` that prints "Migrations may have failed, but continuing‚Ä¶" and starts the app regardless. Live logs show migration 0000 dying on `type "user_role" already exists` (the migration tracking table has 0 rows, so every boot re-runs from scratch and applies nothing).
- **Impact:** Real, observed schema drift: migration `0028` adds enum value `ic_agreement` which is **absent** from the live DB, but `lib/constants.ts:120` and `server/api/routes/onboarding.ts:157-166` insert onboarding steps using it on every new provider registration ‚Äî that INSERT will throw and abort the registration transaction, breaking provider onboarding.
- **Fix:** Make migration failure fatal in production (drop the catch-all continue) or run migrations as a separate gated release step. Reconcile the drizzle migration history against the live schema (which is being maintained out-of-band via `drizzle-kit push`).

#### H6. IDOR ‚Äî any provider can change the status of any invoice
- **Location:** `server/api/routes/invoices.ts:398-425` (`PATCH /:id/status`)
- **Plain English:** Unlike `PATCH /:id`, `DELETE /:id`, and the PDF route ‚Äî which all scope providers by `createdById` ‚Äî this handler only checks the caller is a provider/admin, then updates by `id` with no ownership filter.
- **Impact:** Any provider can mark arbitrary invoices paid/overdue/cancelled, corrupting accounts-receivable and causing revenue loss.
- **Fix:** Add the `createdById` ownership guard consistent with the sibling endpoints.

#### H7. Rate limiter collapses all clients into one shared bucket
- **Location:** `server/api/middleware/rate-limit.ts:19-36`; `server/api/lib/rate-limiter.ts:11,106`
- **Plain English:** `getClientId()` returns the literal string `"unknown-client"` unless `TRUST_PROXY=true` or a Cloudflare header is present. The default deploy has neither (`.env.example` sets `TRUST_PROXY=false`, compose passes nothing), so every request to a path shares one counter. The store is also in-memory per-process.
- **Impact:** A single attacker can exhaust the shared auth bucket (10 req / 5 min) and **lock out all legitimate users** from register/forgot-password/reset/resend-verification/mobile-login.
- **Fix:** Make `TRUST_PROXY` (with the correct trusted-proxy chain) mandatory in Coolify; verify the real proxy header; move the limiter to a shared store (Redis) and key auth limits on email as well as IP.

### MEDIUM

#### M1. IDOR ‚Äî provider can send any draft invoice
- **Location:** `server/api/routes/invoices.ts:373-395` (`POST /:id/send`)
- Same missing `createdById` guard as H6. A provider can finalize/send another provider's or an admin's draft invoice. **Fix:** add the ownership guard.

#### M2. Provider can enumerate the entire customer PII database
- **Location:** `server/api/routes/user-search.ts:21-53` (`GET /search`)
- Any provider (not just admin) can search all `role=customer` users by name/email/phone with a 2-char query. `LIMIT 10` slows it, but substring iteration harvests the full contact DB. **Fix:** scope to customers the provider actually serviced, mask contact fields, rate-limit, or restrict to admin.

#### M3. Company bank details exposed to any authenticated user
- **Location:** `server/api/routes/business-settings.ts:22-25` (`GET /`)
- `GET` uses `requireAuth` (any role) and returns the full row including `bankAccountNumber`, `bankRoutingNumber`, `bankSwiftCode` (`db/schema/business-settings.ts:15-17`). **Fix:** restrict to admin or strip bank fields for non-admins.

#### M4. Admin `/customers` over-fetches password hashes + tax IDs into the API response
- **Location:** `server/api/routes/admin.ts:1001-1017` (`select({ user: users })`)
- Selecting the whole `users` table ships every customer's bcrypt hash, encrypted `taxId`, and `stripeCustomerId` to the admin client. Admin-gated, but hashes should never leave the server (admin XSS, proxy logs, browser memory). **Fix:** whitelist explicit safe columns; audit other `.select({ x: <table> })` patterns.

#### M5. CSV/formula injection in admin exports via unauthenticated guest booking fields
- **Location:** `lib/csv.ts:1-19`; sinks `server/api/routes/admin.ts:918-930,962-972`; source `server/api/routes/bookings.ts:26` (public, unauth)
- `escape()` only handles delimiters, never neutralizes leading `= + - @`. An anonymous attacker submits `contactName=‚Ä¶=cmd|'/c calc'!A1`; when an admin opens `bookings.csv` in Excel/Sheets the formula can execute. **Fix:** centrally prefix any value starting with `= + - @ tab/CR` with a single quote in `generateCSV()` (covers all exports).

#### M6. Webhook idempotency is in-memory only ‚Äî double-payout / duplicate-clawback risk
- **Location:** `server/api/routes/webhooks.ts:27-44,108-111,683`; `server/api/lib/payout-calculator.ts:47-53`; `db/schema/provider-payouts.ts`
- Event dedup uses a per-process `Set` (lost on restart, not shared across instances ‚Äî the code comment admits this). No DB unique constraint on `payments.stripeSessionId` or `providerPayouts.bookingId`, and the read-then-write guards have a race window. A Stripe retry overlapping a slow handler, or a redelivered `dispute.lost`, can insert duplicate payouts/clawbacks (`payout-${id}` idempotency keys differ, so real transfers fire twice). **Fix:** persistent `webhook_events` table keyed by event id (unique); unique constraint on `providerPayouts.bookingId` for standard payouts and on `payments.stripeSessionId`; unique on clawback `originalPayoutId`.

#### M7. `charge.refunded` webhook never claws back provider payout
- **Location:** `server/api/routes/webhooks.ts:182-200`
- Updates only the `payments` row; unlike the admin refund path and `dispute.lost`, it creates no clawback. An out-of-band refund (Stripe dashboard) leaves the provider's payout intact while the platform eats the full refund. **Fix:** mirror the clawback logic in this branch.

#### M8. Destination-charge refunds don't reverse the Connect transfer or refund the app fee
- **Location:** `server/api/routes/admin-payouts.ts:279-291`
- `refunds.create` is called with only `payment_intent` + `amount` ‚Äî no `reverse_transfer` / `refund_application_fee` ‚Äî even though funds were already transferred to the provider's Connect account. The platform absorbs the provider's share. Flag-gated (`DESTINATION_CHARGES`). **Fix:** set `reverse_transfer` and `refund_application_fee` when `chargeType==='destination'`.

#### M9. Provider payout can exceed the service's intended split (`Math.max` floor)
- **Location:** `server/api/lib/payout-calculator.ts:81-92` (also duplicated in `auto-dispatch.ts`, `auto-dispatch-v2.ts`, `admin.ts:307`)
- Two differently-defined rates (service = platform cut, provider = provider share) are reconciled with `Math.max`, turning the default provider rate (7000 = 70%) into a hard payout floor. Any service with a platform cut >30% silently overpays the provider. Latent today because seed services sit at/below the floor. **Fix:** pick one source of truth per booking; only let `provider.commissionRate` override when it's an explicit special arrangement.

#### M10. Pricing time-blocks & earnings windows computed in UTC, not Atlanta time
- **Location:** `server/api/lib/pricing-engine.ts:44-45`; `admin.ts:63`; `provider.ts:404,573`; `auto-dispatch-v2.ts:48-56`
- `getHours()`/local date boundaries run in the container's UTC (no `TZ` set anywhere). An admin's 18:00 ET night surcharge actually fires ~14:00 ET; "today"/"this week" earnings bucket on UTC midnight (8pm ET). Customers mischarged, stats misreported. **Fix:** compute hour/day-of-week in `America/New_York` (Intl with `timeZone`, or set `TZ=America/New_York` in the container) everywhere.

#### M11. Weak, guessable Postgres credentials (`dealer` / `dealer_pass`)
- **Location:** Live `DATABASE_URL` + DB container env; `docker-compose.yml:17,51`
- Password is trivially guessable. **Not** internet-exposed (port not published), but the DB sits on the **shared Coolify bridge network** alongside ~13 other tenant containers ‚Äî any compromised neighbor or app-level SSRF can authenticate to the PII/payments DB. **Fix:** rotate to a long random password in Coolify; keep the DB unpublished and ideally on an isolated network.

#### M12. Server Google Maps key is identical to the browser-exposed `NEXT_PUBLIC` key
- **Location:** Live env `GOOGLE_MAPS_API_KEY_SERVER` == `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY`; used `lib/geocoding.ts:9`, `lib/hooks/use-google-maps.ts:9`
- The "server" key is the same value as the public key baked into client JS (found verbatim in `.next/static` chunks), so it's publicly extractable ‚Äî defeating the purpose of a separate server key. If not strictly referrer/API-restricted in Google Cloud, anyone can run up billing. **Fix:** issue two distinct keys (referrer-restricted browser key, IP/API-restricted server key) and rotate the leaked one.

#### M13. High-severity CVEs in production dependencies
- **Location:** `package.json` ‚Äî `next 16.1.6`, `hono ^4.11.7`, `drizzle-orm ^0.45.1`; transitive `axios 1.13.4` (via twilio)
- `npm audit --omit=dev` reports 6 high advisories on directly-used packages (drizzle SQL-identifier escaping, hono path traversal / middleware bypass / JWT, next request smuggling / Server Actions CSRF / DoS) plus axios SSRF/prototype-pollution. **Fix:** `npm audit fix`; bump next to latest 16.x patch (needs `--force`), hono/drizzle/axios non-breaking; re-audit until high/critical clear.

### LOW

- **L1. Non-idempotent admin confirm-payment** ‚Äî `server/api/routes/admin.ts:571-626`: each call inserts a new confirmed payment and increments trust tier; double-click creates duplicate payments and inflates trust. Check for existing confirmed payment first.
- **L2. Partial refunds recorded as full** ‚Äî `server/api/routes/webhooks.ts:187-198`: first partial refund flips status to `refunded`, blocking later increments and misreporting revenue/refund-rate. Add `partially_refunded` and always update `refundAmount` to `amount_refunded`.
- **L3. Unauthenticated geocoding proxy to paid Google API** ‚Äî `server/api/routes/geocoding.ts:6`: no auth/rate-limit; anonymous loop drives Maps billing. Require auth or `rateLimitStrict`.
- **L4. Public reviews endpoint leaks full customer names** ‚Äî `server/api/routes/reviews.ts:23-53`: unauthenticated `GET /provider/:id` returns reviewers' full account names. Show "First L." instead.
- **L5. LIKE/ILIKE wildcards not escaped** ‚Äî `user-search.ts:44-46` and 6 other search endpoints: not SQLi (params bound), but `%`/`_` broaden matches / force seq scans. Escape `% _ \` with an ESCAPE clause.
- **L6. Direct-to-S3 uploads skip magic-byte + size validation** ‚Äî `onboarding.ts:925-939`, `lib/s3.ts:79-92`: presigned PUT trusts client MIME and has no Content-Length cap; size only checked on the later DB record. Contained (private objects, provider-prefixed keys, restricted MIME). Run `validateFileContent` server-side or add a Content-Length condition.
- **L7. Booking status transitions use unguarded WHERE (TOCTOU)** ‚Äî `auto-dispatch-v2.ts:224`, `provider.ts:140`, `offer-expiry.ts:44`: read-then-write with no expected-status precondition; double-dispatch race. Latent (offer mode default-off) but goes live if H4 is fixed and `DISPATCH_OFFER_MODE` enabled. Use conditional `UPDATE ‚Ä¶ WHERE status=expected ‚Ä¶ RETURNING`.
- **L8. Repo `docker-compose.yml` publishes Postgres 5432 + app 3001 to host** ‚Äî `docker-compose.yml:14,55`: live Coolify deploy does NOT use these mappings (internal-only), but anyone running this compose as-is exposes the DB. Use `expose:`/internal network only.
- **L9. Live Twilio creds + `AUTH_SECRET` readable via `docker exec env`** ‚Äî real Twilio token (SMS spend) and the NextAuth JWT signing secret (forge any session, incl. admin) sit in plaintext container env. Conventional, but consider Coolify secrets/vault and restrict host/docker access; rotate if access is broad.

## 3. Blind Spots ‚Äî Systemic Risks for a Small Team

These are the non-obvious things that will bite a solo founder:

1. **"It works locally" ‚âÝ production.** The single biggest pattern here is **deployment drift**: the custom server with cron/WS isn't what actually runs (H4), migrations silently fail every boot (H5), and prod is running on `.env.example` placeholders (H2/H3). None of these throw a visible error ‚Äî the app boots "green" while payments, email, payouts, and reconciliation are quietly broken. **You cannot trust a successful deploy as proof the system works.** Add boot-time assertions that hard-fail on placeholder secrets and on a missing cron heartbeat.

2. **Fail-open guardrails.** The env validator (`lib/env.ts`) detects every one of these placeholder problems and only `console.warn`s. The migration step catches its own failure and continues. The instinct to "not block the deploy" is exactly backwards for a payments app ‚Äî these should fail closed.

3. **No rate limiting where it matters, and broken where it exists.** The web login has none (H1); the limiter that does exist collapses everyone into one bucket without `TRUST_PROXY` (H7) and is in-memory per-process. A naive fix won't help until the proxy/IP plumbing and a shared store are in place.

4. **Money paths lack DB-level invariants.** Idempotency and ownership are enforced in application code with read-then-write checks (H6, M6, L1, L7) and inconsistent ownership guards (H6, M1). There are **no unique constraints** backstopping `bookingId`/`stripeSessionId`/clawbacks. On a payments system, the database should enforce "one payout per booking," not the app.

5. **Shared-host blast radius.** Your DB is on the shared Coolify network with ~13 other tenant containers, guarded only by `dealer_pass` (M11). A breach of *any* neighbor ‚Äî not just your app ‚Äî reaches your customer/payment data. This is invisible until it isn't.

6. **No monitoring/alerting for silent failures.** Failed emails return `{success:false}` and move on; failed webhooks, dead cron, and migration errors only show in logs nobody reads. There's no Sentry/healthcheck/alert wiring observed. For a 4-person team, **the absence of alerting is itself a top risk** ‚Äî every finding above could run for weeks undetected.

7. **Backup/DR not evidenced.** Nothing in the repo or compose indicates Postgres backups, PITR, or a restore runbook. With one DB container holding all PII/payments and weak creds, a single bad migration (already happening, H5) or volume loss is existential. Confirm Coolify-level backups exist and test a restore.

8. **Dependency rot** (M13) ‚Äî easy to ignore on a small team, but `next`/`hono`/`drizzle`/`axios` all carry high advisories on the exact surfaces you use.

## 4. Quick Wins (do these first)

1. **Rotate the two admin passwords in the live DB now, and remove the `admin123` constant** (C1). Single highest-impact action; takes minutes.
2. **Set real secrets in Coolify and make `lib/env.ts` hard-fail on placeholders in production** (H2 + H3). Fixes payments, email, OAuth, and the forgeable webhook in one change, and prevents recurrence.
3. **Fix the production entrypoint to run the custom server (or add an `instrumentation.ts` hook), and make migrations fatal** (H4 + H5). Restores payouts/reconciliation/realtime and stops silent schema drift breaking onboarding.
4. **Add the missing `createdById` ownership guards to `PATCH /:id/status` and `POST /:id/send`** (H6 + M1) and **add the central CSV formula-escape in `generateCSV()`** (M5). Three small, surgical code edits closing real access-control / injection holes.
5. **Add rate limiting + lockout to the web login and make `TRUST_PROXY` mandatory** (H1 + H7), and **rotate the Postgres password** (M11). Closes brute-force and the weak-credential defense-in-depth gap.
