## What Went Well

### 1. TOCTOU Race Condition Mastery
All three stories correctly used WHERE-clause status guards on UPDATE statements to prevent Time-Of-Check to Time-Of-Use race conditions. This pattern was identified in Epic 9 retro and has now become a consistent practice.

```sql
-- Pattern used across all 3 stories
UPDATE providers SET status = 'onboarding'
WHERE id = ? AND status = 'applied'  -- status guard in WHERE clause
```

### 2. Pure Function State Machine
Winston's architecture decision to extract the state machine (`server/api/lib/onboarding-state-machine.ts`) as a pure function module was validated during E2E testing. The state machine correctly blocked invalid transitions (onboarding -> active, onboarding -> rejected) returning clear error messages.

### 3. Clean Separation of Concerns
- Schema in `db/schema/onboarding-steps.ts` and `db/schema/provider-invites.ts`
- Business logic in state machine module
- Access control in `requireOnboardingComplete` middleware
- Routes cleanly split between `onboarding.ts` (provider-facing) and `admin-providers.ts` (admin-facing)

### 4. Comprehensive Test Coverage
237 tests across the epic, including edge cases for TOCTOU, state machine transitions, middleware gating, and 30-day reapply cooldown.

## What Went Wrong

### 1. emailVerified Gap (HIGH — Bug #1)
The apply endpoint creates users without setting `emailVerified`, but `lib/auth.config.ts:33` requires it for non-admin users. Auto-login after apply works (bypasses authorize), but subsequent logins fail with a misleading "Invalid email or password" error. This was NOT caught by unit tests because they don't exercise the full NextAuth authorize flow.

### 2. UI Gating Incomplete (Bug #3)
Sidebar navigation gates Jobs/Earnings/Invoices for onboarding providers but misses Observations/Inspections/Referrals. The gating was done per-route rather than using a centralized "allowed routes for status" config.

### 3. Admin Onboarding UI Missing (Bug #4)
Story 10-3 implemented admin API endpoints (activate, reject, suspend, reinstate, step review) but no corresponding UI was built. The admin providers page only has generic Edit/Deactivate actions. This was a scope gap in the story requirements.

### 4. Epic 9 Retro Follow-Through: 0/8 Action Items Completed
This is the most significant systemic issue. The previous retrospective identified 8 action items including a CRITICAL XSS security audit. None were completed. The same bug classes (silent .catch, info leakage) that were flagged in Epic 9's code review reappeared in Epic 10's code review.

## Key Learnings

1. **Unit tests don't catch integration gaps**: The emailVerified bug exists at the boundary between the onboarding module and NextAuth. Neither module's tests cover the cross-cutting behavior.
2. **E2E testing on Docker reveals real-world issues**: Running the full stack exposed bugs that mock-based tests inherently miss.
3. **UI gating should be centralized**: Rather than per-route gating, maintain a single config of `allowedRoutesByStatus` to prevent gaps.
4. **Story scope must explicitly call out UI for API endpoints**: Story 10-3's admin endpoints need corresponding admin UI — this should be in acceptance criteria.
5. **Retro follow-through needs enforcement**: Without a tracking mechanism, action items get lost between sprints.
